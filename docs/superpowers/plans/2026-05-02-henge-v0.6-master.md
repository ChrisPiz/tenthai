## Por qué los planes 2–11 están pendientes

Spec contiene detalle suficiente para razonar las fases siguientes, pero traducir cada una a tareas TDD bite-sized antes de cerrar la fase anterior produce planning stale: model IDs reales, formatos de respuesta, y problemas concretos del provider abstraction de fase 1 cambian cómo se escribe fase 2+. Cada plan se escribe al cerrar la fase previa con datos verificados.

## Reglas globales (extraídas del spec)

1. No saltarse fases. Cerrar criterio antes de avanzar.
2. Tests primero por fase. Test que define cierre se escribe antes de implementación.
3. Commits atómicos por sub-fase (1.1, 1.2…). Cada commit deja repo verde.
4. No tocar v0.5 que no se reemplaza. Refactor quirúrgico.
5. Provider abstraction es contrato. `canonical_id` (e.g. `anthropic/opus-4-7`) NO se filtra a code afuera del provider. Solo el provider conoce el `raw_model` del SDK.
6. Diferencias de formato entre Anthropic y OpenAI se absorben dentro del provider.
7. Pricing como dato. Si cambia, bumpear `PRICING_VERSION`.
8. Feature flags default `true`, override por `.env`.
9. Quota/rate-limit error de un provider: log, no abortar si quedan ≥6 frames OK. `status: "quota_error"` por frame.
10. No borrar reportes v0.5. Migration solo para nuevos.

## Riesgos conocidos (del spec)

- `gpt-5` puede no existir al implementar → usar frontier disponible (gpt-5-mini, gpt-4o), documentar elección en código + CHANGELOG.
- Formatos de structured output difieren Anthropic vs OpenAI → normalizar dentro del provider.
- OpenAI streaming + token count: usar non-streaming en v0.6 para simplificar.
- Meta-frame puede early-exit en preguntas válidas → flag `--force-full-run` mitiga.

## Lo que NO se hace en v0.6

Critique cruzado, sensitivity analysis, Gemini, memoria entre reportes, outcome tracking, perfil persistente, modo decisión continua, Obsidian, caching por hash, PDF export, streaming. Todo va a v0.7+.
